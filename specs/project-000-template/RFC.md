### REQ-001: 用户注册

- **描述**：用户通过邮箱 + 密码注册账号
- **验收标准**：
  - 注册成功后，用户收到验证邮件
  - 使用已注册邮箱重复注册，返回 HTTP 409 并附带错误信息
  - 密码长度不少于 8 位，不符合要求返回 HTTP 422
- **优先级**：P0
- **状态**：draft
- **关联**：无

---

### REQ-002: 用户登录

- **描述**：用户通过邮箱 + 密码登录，获取 JWT Token
- **验收标准**：
  - 登录成功返回 access_token 和 refresh_token
  - 连续登录失败 5 次，账号锁定 30 分钟
  - Token 有效期：access_token 15 分钟，refresh_token 7 天
- **优先级**：P0
- **状态**：draft
- **关联**：REQ-001

---

### REQ-003: 用户登出

- **描述**：用户主动登出，使当前 Token 失效
- **验收标准**：
  - 登出后，原 Token 不再能访问需认证的接口
  - 支持「登出所有设备」选项，使该用户所有 refresh_token 失效
- **优先级**：P1
- **状态**：draft
- **关联**：REQ-002

## 非功能需求

### NFR-001: 性能

- API 响应时间 P99 < 500ms（非 IO 密集型接口）
- 系统支持 <N> 并发用户，无性能降级
- 数据库查询单次不超过 100ms（慢查询需加索引或缓存）

### NFR-002: 安全

- 所有密码使用 bcrypt 哈希存储，禁止明文
- API 全面启用 HTTPS，禁止 HTTP 降级
- 敏感数据（Token、密码）不得出现在日志中
- SQL 查询必须使用参数化，禁止字符串拼接

### NFR-003: 可维护性

- 测试覆盖率不低于 80%（核心业务逻辑不低于 90%）
- 所有 public 函数必须有 docstring
- 新功能上线前必须通过 `arch-check.py` 和 `lock-check.py`

### NFR-004: 可用性

- 服务 SLA 目标：99.9%（月停机时间不超过 43 分钟）
- 数据库每日备份，RTO < 4h，RPO < 1h
